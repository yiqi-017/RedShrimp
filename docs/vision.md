# RedShrimp Vision

## 1. 项目定位
RedShrimp 不是新的网页动作 agent，也不是小红书专用浏览器自动化工具。

它的定位是：

> **成为 GenericAgent 面向小红书场景的增长决策层。**

也就是说，GenericAgent 负责“看、点、抓、发、记录”，RedShrimp 负责“判断、建议、约束、复盘”。

RedShrimp 重点回答以下问题：

- 当前账号更适合发什么
- 哪种表达方式更容易获得曝光和互动
- 哪个选题更适合当前阶段目标
- 哪些内容风险偏高，应先修改或放弃
- 发完之后为什么有效或无效，下一轮应如何调整

---

## 2. 当前阶段目标
当前版本优先验证一件事：

> **在不改 GenericAgent 源码、也不引入小红书 MCP 的前提下，RedShrimp 是否能提升内容决策质量。**

因此，第一阶段目标不是把系统做得很大，而是先跑通最小闭环：

1. GenericAgent 抓取小红书内容与账号状态
2. RedShrimp 输出结构化策略建议
3. GenericAgent 执行发帖或跳过
4. GenericAgent 回收结果
5. RedShrimp 输出复盘与下一轮建议

---

## 3. 架构边界

### 3.1 GenericAgent 负责
GenericAgent 继续作为执行层，负责：

- agent loop 与任务编排
- 浏览器扫描与 JS 注入
- 小红书网页操作
- 原始页面信息抓取
- 发布动作执行
- 结果回收与原始数据落盘
- 在合适时机调用 RedShrimp

### 3.2 RedShrimp 负责
RedShrimp 作为策略层，负责：

- 爆款分析
- 选题判断
- 发帖方案生成
- 发前审查
- 发后复盘
- 风险提示
- 下一轮优化建议

### 3.3 小红书网页负责
小红书网页本身是被操作对象，提供：

- 搜索结果
- 帖子详情
- 账号主页
- 发布页面
- 可公开观测的互动指标

---

## 4. 明确不做什么
当前版本刻意不做以下事项：

- 不重写 GenericAgent 的通用能力
- 不改 GenericAgent 源码
- 不额外封装一层网页动作工具库
- 不先建设小红书 MCP
- 不追求一次性覆盖所有内容赛道
- 不在第一阶段引入复杂调度、训练或长期记忆系统

这些都不是永远不做，而是当前阶段不优先。

---

## 5. 为什么现在不先做小红书 MCP
当前 GenericAgent 已经具备：

- 页面读取能力
- JS 精准操作能力
- 表单填写与点击能力
- 网页数据抓取能力
- 多步骤浏览器任务执行能力

因此，现阶段小红书 MCP 并不是能力上的前提条件。

如果过早建设 MCP，会把注意力转移到动作封装、接口稳定性和工程维护上，而当前最关键的问题其实是：

> RedShrimp 给出的策略，是否真的能让内容更容易起量。

所以第一阶段应优先验证“策略层是否有效”，而不是“动作层是否优雅”。

---

## 6. 最小闭环能力
第一版先落 3 个核心能力：

1. 爆款分析
2. 发帖方案生成
3. 发后复盘

三者共同形成可验证闭环：

- 先观察赛道内容
- 再生成适合账号的发帖策略
- 然后发出内容并收集结果
- 最后基于结果迭代下一轮

---

## 7. 与 GenericAgent 的关系
RedShrimp 不替代 GenericAgent，而是被 GenericAgent 调用。

关系可以概括为：

- GenericAgent = 执行器
- RedShrimp = 决策器

典型调用方式：

1. GenericAgent 先从页面抓原始数据
2. 整理为结构化输入
3. 调用 RedShrimp 的对应能力
4. 获取结构化输出
5. 按输出执行 publish / revise / skip
6. 记录结果并再次送回 RedShrimp 复盘

---

## 8. 第一阶段完成标志
当以下条件同时满足时，可视为第一阶段落地完成：

- 已有稳定的输入输出契约
- 已有统一 schema 避免自然语言漂移
- 已有可复用 prompt 模板
- 已有 generate_post_plan / postmortem 的最小样例
- GenericAgent 可以按文档在正确时机调用 RedShrimp
- 可以完成一次“分析 → 生成 → 执行 → 复盘”的闭环

---

## 9. 后续扩展方向
在第一阶段闭环稳定之后，再考虑：

- 引入更强的账号画像与赛道画像
- 沉淀选题库与模板库
- 形成发布时机与标签策略规则
- 建立更系统的实验记录与 AB 对比
- 评估是否需要引入小红书 MCP 作为动作层标准化封装
